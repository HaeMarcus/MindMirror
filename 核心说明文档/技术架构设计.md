# MindMirror 技术架构（多维数据 融合RAG 三次记忆）

三类数据源必须同时工作：
- Flomo HTML（碎片化日常/情绪/想法）
- 自由 MD（复盘/长文/项目记录，命名与内容不固定）
- 记账 CSV（结构化财务流水，列名固定）

目标：本地解析 + 本地索引 + 本地检索 + 云端 Claude（`claude-sonnet-4-6`）生成洞察；输出做”证据归因”——概括性引用证据并标明来源类型与时间范围，不做精确证据片段引用。

---

## 0) 全局约束（给模型的系统规则）
- 必须基于提供的 `source_attribution_context` 与统计摘要生成洞察；不确定要明确说不确定
- 【证据归因】必须概括性引用证据，并指出使用了哪些来源（Flomo HTML/MD/CSV）与时间范围（有则写；无则写”未标注/推断弱”）
- 不输出精确引用：不出现 chunk_id、行号、原文片段定位；只做”来源+时间+概括”
- 多用户：通过昵称（nickname）隔离用户数据，无需登录系统；所有数据与索引落本地
- 提供”清除数据”功能（按用户隔离，清除该用户的对话历史、上传文件、记忆）

---

## 1) 三类数据源（Sources）

### 1.1 Flomo HTML（强结构半强语义）
输入：Flomo 导出的 HTML（通常包含卡片/时间/正文）

**Flomo HTML 解析要点（用户给的案例）**
- 每条 memo：`div.memo`
- 时间：`.memo .time`，如 `2025-12-31 20:09:17`
- 正文：`.memo .content` 内多个 `<p>`
- 标签：正文中 `#xxx`
- 图片：`.memo .files img`（可忽略内容）

切片：
- 默认”一条 memo 一个 chunk”
- 超长 memo 再按段落/长度拆分（max_len=500字，min_len=200字，匹配 bge-small-zh-v1.5 的 512 token 有效窗口）

用途：
- 捕捉高频主题、情绪波动、近期关注点、行为习惯
- 在洞察里支撑“日常层面的微观证据”（但仍只来源级归因）

---

### 1.2 自由 MD（弱结构，鲁棒优先）
输入：任意 md 文件（命名、内容自由）

解析：
- 保留纯文本 `raw_text`
- 提取结构特征（若有标题）：
  - 按 `# / ## / ### / ####` 切 section
  - 否则按空行分段 + 长度窗口切（max_len=500字，匹配 embedding 模型有效窗口）

时间范围推断（非强依赖，按优先级尝试）：
1. 优先读 frontmatter（若有 `date/month/year`）
2. 从文件名提取年份（如 `2022年度总结.md` → year=2022，`月度复盘.md` → 无）
3. 从正文找日期模式（`YYYY-MM`、`YYYY年M月`、`M月`）
4. 若只出现”本月/3月”且缺年份：可用文件 mtime 弱推断年份，标 `confidence=low`
5. 推断失败：`time_range=null`，仍可索引与检索

用途：
- 承载“叙事型证据”：阶段性目标、长期动机、项目推进、复盘反思等
- 在洞察里支撑“中观/宏观总结”

---

### 1.3 钱迹 CSV（列名固定，强统计能力）
输入：钱迹 CSV（列名固定不变）

完整列名（共 18 列）：
`ID, 时间, 分类, 二级分类, 类型, 金额, 币种, 账户1, 账户2, 备注, 已报销, 手续费, 优惠券, 记账者, 账单标记, 标签, 账单图片, 关联账单`

MVP 核心列（8个）：`时间, 分类, 二级分类, 类型, 金额, 币种, 备注, 标签`
可忽略列（10个）：`ID, 账户1, 账户2, 已报销, 手续费, 优惠券, 记账者, 账单标记, 账单图片, 关联账单`

注意：时间格式为 `YYYY/MM/DD HH:MM`（用 `/` 分隔，如 `2025/12/31 15:53`）

解析：
- 行级结构化解析（核心列）
- 强制校验核心列名完整性（缺列则报错，避免脏数据）

切片（分层策略）：
1) `ledger_row`：每行存入数据库但**不进行向量化**（faiss_id=NULL），避免大量低语义信号的行 chunk 污染索引
2) `ledger_month_summary`：按月聚合，生成自然语言描述的月度摘要（收支、结余、Top5 分类、细分开销），**参与向量化**
3) `ledger_category_summary`：按分类聚合，生成含交易次数、金额、常见备注的分类摘要，**参与向量化**
4) `ledger_overall_summary`：整份 CSV 的全局概览（时间跨度、总收支、Top5 支出类别），**参与向量化**

设计理由：
- 一份 3000 行 CSV 只产生约 20-40 个高质量摘要向量（而非 3000+ 个弱语义向量），与 Flomo/MD 数据源的向量数量保持均衡
- 摘要文本使用自然语言描述（如”在2025年12月，你的总支出为8000元...”），embedding 模型能更好地理解消费语义

用途：
- 支撑”量化洞察”：消费结构、月度结余、社交支出频率、房租等大额支出影响
- 让模型的建议更可执行（预算/习惯干预）

## 2) 统一中间表示（Normalization）

### 2.1 Document
- `doc_id`: hash(source_type + file_path + mtime + size)
- `source_type`: `flomo_html | review_md | ledger_csv`
- `source_name`: 原文件名
- `time_range`: 
  - HTML：来自 memo 时间的 min/max（若可得）
  - MD：推断得到则填；否则 null
  - CSV：从 `时间` 列 min/max 得出
- `raw_text`: 用于切片的纯文本（CSV为拼接文本或可为空，主要靠行/摘要chunk）
- `metadata`: JSON（推断置信度、解析器版本、字段映射等）

### 2.2 Chunk
- `chunk_id`, `doc_id`
- `chunk_type`:
  - HTML：`memo`
  - MD：`md_section`
  - CSV：`ledger_row`（仅存储）| `ledger_month_summary` | `ledger_category_summary` | `ledger_overall_summary`（后三者参与向量化）
- `content`: embedding 文本
- `metadata`: `source_type/source_name/date_or_period/heading_path/category/...`

## 3) 本地索引层（Indexing）
- Embedding：本地模型 `bge-small-zh-v1.5`（512维，max_seq_length=512 tokens，约 300-400 中文字符有效窗口）
- Vector DB：**Per-user FAISS 索引**，每个用户独立的 `IndexFlatIP` 文件（`data/faiss_index/{hash(nickname)}.faiss`），入库前 L2 归一化（实现 cosine similarity）
- 元数据：SQLite
  - `documents`（含 user_id 字段）
  - `chunks`（faiss_id 仅在用户索引内唯一，跨用户可重复）
  - `messages`（短期对话）
  - `memory`（中期摘要、长期画像、触发计数器）
  - `feedback`（用户反馈）

用户隔离：
- FAISS 搜索直接在该用户自己的索引上执行，不会触碰其他用户的向量
- SQLite 查询通过 user_id + JOIN 做纵深防御
- 清除数据时同步删除用户的 FAISS 索引文件

增量策略：
- doc 变更才重建该 doc 的 chunks 与向量
- 重复上传同一文件会先删除旧数据再重新入库

---

## 4) 检索与融合（RAG，三源协同）
每次用户提问：

1) Query embedding → 在**该用户的 FAISS 索引**中搜索 topK（当前 K=12）
2) Source-aware rerank（基于关键词规则，boost=1.3x）
- 财务/消费/预算相关关键词 → 提升 CSV 摘要 chunks
- 复盘/总结/目标/项目 → 提升 MD section
- 情绪/关系/习惯/自我评价/最近状态 → 提升 HTML memo
- 所有 summary 类型 chunks 额外获得 1.1x boost
3) Evidence Compression（本地规则化压缩，不额外调用 LLM）
- 输入：topK chunks（来自三源）
- 压缩策略：
  - 按 source-aware rerank 分数排序
  - 去重（同源同时段的相似 chunk 合并）
  - 每个 source_type 取 top chunks
  - 截断至 token 上限（建议 2000 tokens）
- 输出：`source_attribution_context`（给 LLM 的”来源级要点包”）

## 5) “证据归因”上下文（给 LLM 的核心输入）
推荐用 JSON（模型更稳，也方便前端隐藏展示）。

`source_attribution_context` 建议结构：
- `sources`: [
  - {
    - `source_type`: `flomo_html`,
    - `source_name`: "...html",
    - `time_range`: "2026-02-01~2026-02-28" 或 null,
    - `confidence`: "high|medium|low",
    - `themes`: ["社交回避", "自尊", "学习推进", ...],
    - `summary_bullets`: ["...","..."]
  },
  - {
    - `source_type`: `review_md`,
    - `source_name`: "...md",
    - `time_range`: null/推断值,
    - `confidence`: "...",
    - `themes`: ["科研推进","数分学习",...],
    - `summary_bullets`: ["...","..."]
  },
  - {
    - `source_type`: `ledger_csv`,
    - `source_name`: "...csv",
    - `time_range`: "2026-02-01~2026-02-28",
    - `confidence`: "high",
    - `themes`: ["社交支出","房租","红包收入",...],
    - `summary_bullets`: ["本月收入主要来自…","支出集中在…",...]
  }
]

重要：如果某个来源本轮没命中（检索不到或相关性低），就在 `sources` 里省略它，LLM 也必须在归因段落里“不提该源”。

## 6) 记忆系统（三层，基于计数器触发）

### 6.1 短期记忆
- 最近 **6 轮**对话（12 条消息）直接发送给 LLM
- 保持即时上下文连贯，降低 API token 消耗

### 6.2 滚动摘要（中期记忆）
- 容量：**500 字以内**，信息密集的用户档案备忘录
- 触发：**每 5 轮**对话更新一次
- 触发机制：在 memory 表中存储 `_summary_updated_at_round` 计数器，当 `当前轮数 - 上次更新轮数 >= 5` 时触发，不依赖精确取模
- 更新逻辑：取最近 12 条消息 + 旧摘要 → Claude 生成新摘要（保留仍成立的发现，融入新主题，删除过时内容）

### 6.3 用户画像（长期记忆）
- 格式：JSON `{ key_facts, themes, values, goals, patterns, risks }`
- 触发：**每 20 轮**对话独立触发更新（不嵌套在摘要更新内）
- 触发机制：memory 表中存储 `_profile_updated_at_round` 计数器，与摘要触发完全解耦
- 更新逻辑：取当前画像 + 滚动摘要 → Claude 谨慎更新（保留稳定特征，缓慢增加新发现）
- 存储内容示例：
  - `key_facts`：用户反复强调或明确告知的客观重要信息（如"叫小明，在北京工作"、"有一个姐姐"、"童年在农村长大"、"2024年经历了裁员"）
  - `themes`：用户长期关注的主题（如"职业转型"、"亲密关系"、"自律养成"）
  - `values`：从对话中推断的核心价值观（如"追求自由"、"重视成长"）
  - `goals`：用户明确或隐含的目标（如"年底前完成副业项目"）
  - `patterns`：反复出现的行为模式（如"压力大时倾向消费补偿"、"拖延后自我否定"）
  - `risks`：需要关注的风险点（如"长期睡眠不足"、"社交回避倾向加重"）

### key_facts 收录原则
- 只收录用户**明确告知**或**在多次对话中反复提及**的客观重要信息
- 不过度收集：只记录对理解用户至关重要的信息，而非所有提及的细节
- 已有的 key_facts 除非用户明确纠正，否则永远保留（防止重要记忆丢失）
- 典型收录范围：姓名/昵称、职业身份、家庭关系、重大人生事件、童年记忆、重要人物
- 不收录：一次性提到的琐碎细节、推测性信息、未经确认的内容

### 更新原则
- 记忆更新只能基于对话 + 来源级要点（避免把幻觉写进 profile）
- profile 更新频率低、要求稳定，只记录跨多次对话反复确认的特征
- key_facts 的保留优先级最高，即使其他字段被精简，key_facts 也不应被删除
- 计数器机制确保即使消息被删除或服务重启，触发逻辑仍然可靠

## 7) LLM 推理层（Claude Sonnet）
输入（最小必要上传）：
- `question`（用户当前问题）
- 最近 6 轮对话历史（短期记忆，12 条消息）
- `user_profile`（长期画像 JSON）
- `rolling_summary`（滚动摘要，500 字以内）
- `source_attribution_context`（核心，来源级压缩要点，每源最多 5 个 chunk，总计 ≤ 2000 tokens）

输出（三段固定）：
1)【核心洞察】一句话定性，直指核心问题
2)【模式识别】从整体上把握用户的行为模式，找出反复出现的主题，多维度交叉分析
3)【证据归因】概括性引用证据 + 标明来源类型与时间范围

---

## 8) API（保留最小闭环）
- `POST /api/ingest`
  - 上传 html/md/csv
  - 自动完成：解析 → 切片 → 向量化 → 入库（合并原 ingest + index/build）
  - 返回：doc_id、解析条数、时间范围（若能得出）、置信度

- `POST /api/chat`
  - 检索→压缩→LLM→写入 messages / 更新 summary/profile（触发式）
  - 返回：四段洞察 + `source_attribution`（结构化、可选展示）

不出现：具体行号、具体段落定位、chunk_id、原文定位链接
